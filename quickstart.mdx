---
title: "Quickstart"
description: "See recognition and a Checkout Session work in test mode in a minute, on any platform, then add Ringup to your agent."
---

Ringup adds identity and payments to a voice agent: a returning caller is recognized by phone
number and pays with a saved card, settling on the merchant's own processor, with no card number
ever spoken. This quickstart shows the two moves everything is built on, in test mode, then points
you to the integration for your platform.

Test mode runs on a shared sandbox with ready-made merchants and callers, so there is nothing to
sign up for and nothing to connect between you and a first working response. See
[Testing](/concepts/testing) for the fixtures.

<Steps>
  <Step title="Recognize a caller">
    Ringup recognizes a caller from their phone number, which arrives the instant the call
    connects. Call `identify` for the sandbox demo caller:

    ```bash
    curl -X POST https://ringup.dev/api/tools/identify \
      -H "X-Caller-Number: +14155550142"
    ```

    ```json
    { "known": true, "name": "Alex", "card": { "brand": "VISA", "last_4": "5858" } }
    ```

    That is the identity layer: at the start of the call your agent greets a returning caller by
    name and can offer their saved card, with no card number spoken. See
    [Caller and merchant identity](/concepts/identity).
  </Step>
  <Step title="Create a Checkout Session">
    When the order is ready to pay, create a Checkout Session. Ringup resolves whether payment is
    required from the merchant's own rule and returns it to you:

    ```bash
    curl -X POST https://ringup.dev/api/tools/create_checkout \
      -H "Content-Type: application/json" \
      -H "X-Ringup-Merchant: tonys-pizza" \
      -d '{ "amount_cents": 2500 }'
    ```

    ```json
    { "id": "cs_d00993cf0f3e", "payment_required": "optional" }
    ```

    The Checkout Session is the object every payment turns on: `payment_required` is `required`,
    `optional`, or `none`, and the charge settles on the merchant's own processor. See
    [How it works](/concepts/how-it-works).
  </Step>
  <Step title="Bring your own API key">
    The calls above run against the open demo sandbox with no key, so you can try them as-is. For
    your own merchants and callers you authenticate with your Ringup API key: create one in the
    dashboard and send it as `Authorization: Bearer <YOUR_RINGUP_KEY>`. See
    [Authentication](/authentication) for test versus live keys and where the key lives in each
    integration.
  </Step>
  <Step title="Add it to your agent">
    Both moves live inside your voice agent. Pick the integration for your platform:

    - **Vapi**: a [native handoff](/integrations/vapi/native), wired by one CLI command, keeps the
      caller in your agent's own voice throughout.
    - **Every other platform** (Retell, ElevenLabs, Deepgram, LiveKit, OpenAI Realtime, Pipecat,
      Twilio ConversationRelay): a [call transfer](/concepts/call-transfer), one integration you
      reuse everywhere. Pick your platform in the Integrations section.
  </Step>
</Steps>

<Note>
On Vapi you can also hear a fully wired demo in about a minute. `npx ringup-vapi-init test` builds
a demo squad in your own Vapi org, backed by the shared sandbox, with a demo caller and a saved
card. The full CLI lifecycle (`test`, `stage`, `go-live`, `rollback`) is in the
[Vapi guide](/integrations/vapi/native).
</Note>
