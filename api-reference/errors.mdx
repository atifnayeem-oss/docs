**One envelope, every endpoint.** The same object comes back from every call, so you write one
parser. A declined charge is no exception: it arrives as a `402` in this same shape. Branch on
`type` first, then refine on `code`.

```json
{
  "error": {
    "type": "processor_error",
    "code": "card_declined",
    "audience": "caller",
    "decline_code": "insufficient_funds",
    "advice_code": "try_again_later",
    "message": "Charge declined by the issuer.",
    "agent_message": "That card was declined for insufficient funds. Would you like to try a different card?",
    "display_message": "That card was declined for insufficient funds.",
    "agent_action": "offer_different_card",
    "doc_url": "https://docs.ringup.dev/api-reference/error-codes#card_declined",
    "charge": "chg_77386e2f",
    "request_id": "req_8f3c1d2ab4e9"
  }
}
```

## Attributes

<ResponseField name="type" type="string">
  The broad class. Branch on this first: it is a small, stable set, listed below.
</ResponseField>
<ResponseField name="code" type="string">
  The specific, machine-readable reason. Every code is catalogued in [Error codes](/api-reference/error-codes).
</ResponseField>
<ResponseField name="audience" type="string">
  Who this error is for: `developer` or `caller`. It decides whether the three spoken fields
  carry anything. See [Handling errors](/api-reference/handling-errors).
</ResponseField>
<ResponseField name="message" type="string">
  A developer-facing explanation. Written for your logs. **Never speak this to a caller.**
</ResponseField>
<ResponseField name="agent_message" type="string | null">
  The sentence your voice agent can say, verbatim. Null on every `developer` error.
</ResponseField>
<ResponseField name="display_message" type="string | null">
  The same fact written to be read rather than spoken, for a screen or an SMS. Null on every
  `developer` error.
</ResponseField>
<ResponseField name="agent_action" type="string | null">
  The next payment move, when there is one: `retry_card`, `offer_different_card`, or
  `send_link`. Null whenever the next move is not ours to choose.
</ResponseField>
<ResponseField name="advice_code" type="string">
  Our guidance on retrying: `try_again_later`, `confirm_card_data`, or `do_not_try_again`.
  Present on every error.
</ResponseField>
<ResponseField name="charge" type="string">
  The charge this error relates to, when there is one.
</ResponseField>
<ResponseField name="param" type="string">
  The parameter at fault, when one field caused the error.
</ResponseField>
<ResponseField name="decline_code" type="string">
  The issuer's reason, on a decline. For your logs. Sensitive reasons are deliberately not
  reflected in `agent_message`.
</ResponseField>
<ResponseField name="processor_code" type="string">
  The processor's own code, unmodified, on a `processor_error`.
</ResponseField>
<ResponseField name="doc_url" type="string">
  Links straight to that code's entry in [Error codes](/api-reference/error-codes).
</ResponseField>
<ResponseField name="request_id" type="string">
  Unique per request, also returned as the `Ringup-Request-Id` header on **every** response.
  Quote it in support requests: it matches our logs exactly.
</ResponseField>

## Error types

Branch on `type` first. It is seven values and it changes rarely; `code` refines within it.

<ResponseField name="invalid_request_error" type="error type">
  The request is malformed, or names something that does not exist. Fix the request; retrying it unchanged will fail the same way.
</ResponseField>
<ResponseField name="authentication_error" type="error type">
  The API key is missing, malformed, or revoked. Check the `Authorization` header.
</ResponseField>
<ResponseField name="permission_error" type="error type">
  The key is valid but is not allowed to act on this resource.
</ResponseField>
<ResponseField name="processor_error" type="error type">
  The merchant's payment processor declined the payment or could not be reached. Carries `processor_code`, and `decline_code` when the issuer gave a reason.
</ResponseField>
<ResponseField name="idempotency_error" type="error type">
  An idempotency key was reused with different parameters.
</ResponseField>
<ResponseField name="rate_limit_error" type="error type">
  Too many requests. Back off and retry.
</ResponseField>
<ResponseField name="api_error" type="error type">
  A fault on Ringup's side. Safe to retry.
</ResponseField>

Every endpoint in this reference lists the specific codes it can return, so you never have to
hold the whole catalog in mind to know what one call can do.
