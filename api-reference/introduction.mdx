Ringup has two tiers, like a hosted-checkout provider. Most integrations use the high-level
**Checkout Session** (create one, transfer the call, reconcile on the webhook); see
[Checkout Session](/concepts/checkout-session). The low-level endpoints below are the **Payment**
tier, for platforms that own their entire conversation loop and drive the payment operations
themselves. Use a Checkout Session unless you have a reason not to.

<Note>
**Stability.** The request and response shapes on this page are stable. The API is versioned in
the base path (`/v1`): within a version, changes are additive (new optional fields only), and any
breaking change ships under a new version. Pin the version and upgrade on your own schedule. See
the [Changelog](/changelog) for what has shipped.
</Note>

## Base URL

```text
https://api.ringup.dev/v1
```

All five tools are `POST` endpoints with JSON bodies.

## Authentication

Every request authenticates with your Ringup API key as a Bearer token:

```bash
Authorization: Bearer <YOUR_RINGUP_KEY>
```

The public demo sandbox is open (no key) so you can try the endpoints and the
[Quickstart](/quickstart) before signing up. For your own merchants and callers, use your key.
See [Authentication](/authentication) for test versus live keys and how to keep the key safe.

## Identity rides headers, not arguments

The caller and the merchant are resolved server-side from three headers your platform
templates in at call time. The model never types either identity, so neither can be
mistyped, hallucinated, or prompt-injected. See [Caller Identity](/concepts/identity) and
[Merchant Identity](/concepts/merchant-identity).

| Header | Value | Purpose |
|---|---|---|
| `X-Caller-Number` | The caller's number, E.164 (platform caller-ID variable) | Who is calling |
| `X-Ringup-Merchant` | The dialed number, E.164, or your explicit merchant id | Which business they called |
| `X-Call-Id` | The platform's call id | Charge idempotency: retries can never double-charge |

<Warning>
Web and dashboard test calls carry no real caller ID, so the caller-ID template resolves to
nothing. Pin a test caller explicitly for those; real inbound phone calls resolve
everything natively.
</Warning>

## Idempotency

Charges are deduplicated server-side against the call id and amount. A platform retry, a
model retry, or a duplicated tool call returns the original payment result instead of
charging twice. This is a server guarantee, not a client convention.

## Webhooks

Payment outcomes also arrive out of band as signed events (`checkout.completed` and the rest of
the catalog). Add your endpoint in the dashboard under **Settings, then Webhooks**, and copy the
signing secret. See [Webhooks](/concepts/webhooks).

## Test mode

The shared sandbox answers the same five tools, keyless, at `https://ringup.dev/api/tools`, with
ready-made fixtures (merchants with different policies, a caller with a saved Visa). See
[Testing](/concepts/testing).
