Add card-on-file payments to a Retell agent by call transfer: when the order is confirmed, your
agent creates a **Checkout Session** and transfers the live call to it, like redirecting to a
hosted web checkout. If the model is new to you, read
[Call transfer](/concepts/checkout-session) first.

## Prerequisites

- A Retell agent you already run, with a phone number (see the warning below).
- Your Retell API key, and a Ringup [API key](/authentication).

## Set up

<Tabs>
<Tab title="Using the dashboard">

<Steps>
  <Step title="Connect Retell, once for your account">
    In the Ringup dashboard, open **Integrations**, then **Connect Retell**, and paste your Retell
    API key. Ringup uses it to list your agents and provision the payment objects, and never
    stores it. One connection covers every merchant you run.
  </Step>
  <Step title="Pick the agents that take payments">
    Select the agents that answer merchant calls. Ringup registers `identify` (called at the
    greeting), `create_checkout`, and the `transfer_to_payment` tool on each; paste the one prompt
    line the dashboard shows. No per-merchant step: the merchant resolves from the dialed number,
    and activates when it connects its processor.
  </Step>
  <Step title="Add your webhook endpoint">
    Under **Settings, then Webhooks**, add your HTTPS endpoint and copy the signing secret. Ringup
    posts a signed `checkout.completed` there for every payment. See [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Rehearse on the rehearsal number, then go live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, order, and pay
    with the test card (Visa ending 5858): no real charge. When it reads a confirmation code back,
    click **Go live**.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

<Steps>
  <Step title="Recognize the caller at the greeting">
    Register `identify` as a custom function (it takes no arguments; the caller's number comes
    from the call) and call it at the greeting, not at payment. For a returning caller it returns
    their name and saved card, so your agent greets them by name. See
    [Recognize at the start of the call](/concepts/identity).
  </Step>
  <Step title="Add the Ringup checkout function">
    Register a custom function on your agent that creates a Ringup Checkout Session, authenticated
    with your Ringup API key, and add one prompt line: *when the caller has confirmed their order,
    call `create_checkout` with the order total.*

    ```jsonc
    {
      "type": "custom",
      "name": "create_checkout",
      "description": "Create a Ringup Checkout Session for the order and return where to transfer the call.",
      "url": "https://api.ringup.dev/v1/checkouts",
      "method": "POST",
      "args_at_root": true,
      "headers": {
        "Authorization": "Bearer <YOUR_RINGUP_KEY>",
        "X-Caller-Number": "{{user_number}}"        // filled by Retell, never by the model
      },
      "parameters": {
        "amount_cents":    { "type": "number", "description": "Order total in cents" },
        "success_message": { "type": "string", "description": "Optional warm closing line Ringup reads before ending the call" },
        "order_id":        { "type": "string", "description": "Your processor order id, if you have one" }
      },
      "response_variables": { "ringup_transfer_uri": "transfer_to" },  // fills the transfer destination server-side
      "speak_during_execution": true,
      "execution_message_type": "static_text",
      "execution_message_description": "One moment, putting your payment through now."
    }
    ```
  </Step>
  <Step title="Wire the transfer">
    Add a `transfer_call` tool whose destination is filled from the `ringup_transfer_uri` dynamic
    variable, never a value the model types, and cold-transfer over SIP REFER:

    ```jsonc
    {
      "type": "transfer_call",
      "name": "transfer_to_payment",
      "transfer_destination": { "type": "predefined", "number": "{{ringup_transfer_uri}}" },
      "transfer_option": { "type": "cold_transfer", "cold_transfer_mode": "sip_refer" }
    }
    ```

    The session travels inside the transfer address itself, as a signed token in the SIP URI;
    custom SIP headers do not survive Retell's transfer leg.
  </Step>
  <Step title="Add your webhook endpoint">
    In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy
    the signing secret. Your server verifies the signature and handles `checkout.completed`. See
    [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Test it, then go live">
    Point the `create_checkout` function's `url` at the keyless sandbox,
    `https://ringup.dev/api/tools/create_checkout` (or keep the prod URL with your `rk_test_` key).
    Call your agent and order: the test caller (`+14155550142`) pays on the saved Visa ending 5858;
    a first-time caller is texted a link and enters `4111 1111 1111 1111`. Expect a confirmation
    code read back and a test-mode `checkout.completed`. Swap the URL back to the live endpoint to
    go live.
  </Step>
</Steps>

</Tab>
</Tabs>

<Warning>
Transfer to the SIP URI the Checkout Session returns, not to a PSTN number, and from a number
Retell can move a call from: numbers purchased through Retell and numbers imported over an elastic
SIP trunk both work. For an imported number, enable call transfer (SIP REFER) on the trunk at your
telephony provider, or the transfer is refused. A call that reaches the agent by dial-to-SIP-URI (a
registered call) cannot use Retell's transfer at all; put agents that take payments on a real
number.
</Warning>
