Add card-on-file payments to a Pipecat voice pipeline two ways. **Hosted** transfers the live call
to a Ringup payment line at payment time, so Ringup runs the whole payment. **In your agent** keeps
the caller in your pipeline and calls Ringup's tools from your own code (recognize the caller, charge
the saved card), with no transfer at all. A recognized caller never reads a card aloud, so no card
data touches your pipeline either way.

## Prerequisites

- A running Pipecat pipeline with telephony (a Twilio/Telnyx/Plivo/Exotel WebSocket serializer,
  or Daily PSTN/SIP dial-in).
- The telephony account that owns the inbound call leg, and a Ringup [API key](/authentication).

## Set up

<Tabs>
<Tab title="Using Dashboard">

Your pipeline creates a Checkout Session and transfers the live call to it, like redirecting to a
hosted web checkout. Ringup takes payment on its own line and returns the caller if you ask it to.

<Note>
The transfer is code in your pipeline, so there is no dashboard variant of the wiring itself. Your
Ringup account side (API keys, the webhook endpoint, go-live) lives in the dashboard.
</Note>

<Steps>
<Step title="Recognize the caller at the greeting">

Call `identify` from your pipeline at the greeting, not at payment (a plain HTTP call, like the
checkout call below). The caller's number is available the instant the call connects; for a
returning caller `identify` returns their name, so your agent greets them by name. The cards it
can pay with come later, from the Checkout Session.
See [Recognize at the start of the call](/concepts/identity).

</Step>
<Step title="Create a Checkout Session when the order is confirmed">

Register a function on your LLM (`FunctionSchema`) that the model calls once the order is confirmed,
and in its handler `POST` to Ringup with your API key. If `payment_required` is `none`, do not
transfer: finish the call normally. Otherwise transfer to the returned `transfer_to`.

```python
async def create_checkout(params: FunctionCallParams):
    r = await http.post(
        "https://api.ringup.dev/v1/create_checkout",
        headers={"Authorization": f"Bearer {RINGUP_KEY}"},
        json={
            "merchant_id": merchant_id,
            "caller_phone": caller_number,
            "order_id": params.arguments.get("order_id"),          # the total is read from it
            "platform_call_id": call_id,                           # optional
            "return_to": "sip:your-bridge@your-telephony",         # optional: bring the caller back
            "success_message": "You are all set, your order will be ready shortly.",  # optional
        },
    )
    await params.result_callback(r.json())
```

```jsonc
{
  "checkout_session_id": "cs_x9f2",                  // the correlation token
  "payment_required": "required",                    // "required" | "optional" | "none"
  "amount_cents": 1900,                              // resolved from the order
  "currency": "USD",
  "amount_source": "order",                          // "order" | "request"
  "transfer_to": "sip:eyJjIjoiKzE0MTU1NTUwMTQyIiwiYSI6MTY1MCwibSI6Im1jaF80ZjJjOWExYjdlM2QifQ.Xr7QpN2vK9sLd4TfA1mYc8BhE3wZ@transfer.ringup.dev",  // null unless payment is required
  "expires_at": "2026-07-21T18:35:12Z",
  "cards": [],                                       // saved cards chargeable at this merchant
  "default_card_id": null,
  "metadata": null
}
```

</Step>
<Step title="Transfer on your transport">

Have the agent speak one short bridge line ("One moment, connecting you to secure payment"), then
transfer the caller on your telephony. On a Twilio transport, redirect the call to TwiML that dials
the Ringup SIP URI, carrying the `checkout_session_id`:

```xml
<Response>
  <Dial>
    <Sip>sip:eyJjIjoiKzE0MTU1NTUwMTQyIiwiYSI6MTY1MCwibSI6Im1jaF80ZjJjOWExYjdlM2QifQ.Xr7QpN2vK9sLd4TfA1mYc8BhE3wZ@transfer.ringup.dev</Sip>
  </Dial>
</Response>
```

On Daily PSTN/SIP, use the equivalent SIP transfer to the same URI.

</Step>
<Step title="Register your webhook endpoint">

In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy the
signing secret. Your code verifies the signature and handles `checkout.completed`. See
[Webhooks](/concepts/webhooks).

</Step>
<Step title="Test it, then go live">

Point your `POST` at the keyless sandbox, `https://ringup.dev/api/tools/create_checkout` (or keep
the prod URL with your `rk_test_` key), call your agent, and order: the test caller
(`+14155550142`) pays on the saved Visa ending 5858; a first-time caller is texted a link and
enters `4111 1111 1111 1111`. Expect the payment line to read a confirmation code and a test-mode
`checkout.completed` at your endpoint. Swap the URL back to the live endpoint to go live.

</Step>
</Steps>

</Tab>
<Tab title="Using Tools">

Keep the caller in your pipeline and call Ringup's tools straight from your function handlers with
plain HTTP. Recognize the caller at the greeting, then charge their saved card when the order is
confirmed. No transfer, no payment line.

<Steps>
<Step title="Recognize the caller at the greeting">

Call `identify` at the greeting, not at payment. Pass `caller_phone`, available the instant the
call connects. For a returning caller it returns their name, so your agent greets them by name and
skips questions it already has answers to. An unknown caller comes back `known: false`, which is
an ordinary first call. See [Recognize at the start of the call](/concepts/identity).

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/identify \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "caller_phone": "+14155550142" }'
```

```python Python
caller = requests.post(
    "https://api.ringup.dev/v1/identify",
    headers={"Authorization": "Bearer <YOUR_RINGUP_KEY>"},
    json={"caller_phone": caller_number},
).json()
# caller["known"] tells you whether to greet by caller["consumer"]["first_name"]
```
</CodeGroup>

</Step>
<Step title="Open a Checkout Session when the order is confirmed">

`POST` to `create_checkout` with the merchant and the caller. Ringup resolves the total (from the
merchant's order when you pass `order_id`), applies the merchant's
[payment policy](/concepts/payment-policy), and returns `cards`: the caller's cards chargeable at
this merchant. On `payment_required: "none"` say nothing about payment and finish the call.

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/create_checkout \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
    "merchant_id": "tonys-pizza",
    "caller_phone": "+14155550142",
    "order_id": "ord_7f21"
  }'
```

```python Python
checkout = requests.post(
    "https://api.ringup.dev/v1/create_checkout",
    headers={"Authorization": "Bearer <YOUR_RINGUP_KEY>"},
    json={
        "merchant_id": merchant_id,
        "caller_phone": caller_number,
        "order_id": order_id,  # or amount_cents, when there is no order
    },
).json()
# checkout["payment_required"] == "none" means say nothing about payment
```
</CodeGroup>

</Step>
<Step title="Charge the card the caller picked">

Read one card from `cards` back by brand and last four with the amount. After the caller agrees,
`POST` to `charge` with the session and that card's `instrument_id`. Ringup charges it off-session
on the merchant's own processor. A decline comes back as a `402` in the standard
[error envelope](/api-reference/errors), carrying the sentence your agent can speak.

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/charge \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "checkout_session_id": "cs_x9f2", "instrument_id": "ins_4b7e" }'
```

```python Python
res = requests.post(
    "https://api.ringup.dev/v1/charge",
    headers={"Authorization": "Bearer <YOUR_RINGUP_KEY>"},
    json={
        "checkout_session_id": checkout["checkout_session_id"],
        "instrument_id": card["instrument_id"],  # the card the caller agreed to
    },
)
# a decline is a 402: speak res.json()["error"]["agent_message"]
charge = res.json()
# read charge["order_id"] back only when charge["status"] == "completed"
```
</CodeGroup>

</Step>
<Step title="Text a link when there is no card to offer">

When `cards` is empty (or the charge declined), `POST` to `send_link`. The session is the only
input, and Ringup texts a secure pay link. Check `sent`: `false` means a duplicate send returned the
same link without texting again. See [Send Link](/concepts/send-link).

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/send_link \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "checkout_session_id": "cs_x9f2" }'
```

```python Python
link = requests.post(
    "https://api.ringup.dev/v1/send_link",
    headers={"Authorization": "Bearer <YOUR_RINGUP_KEY>"},
    json={"checkout_session_id": checkout["checkout_session_id"]},
).json()
# link["sent"] is False on a duplicate send: no new text went out, so do not promise one
```
</CodeGroup>

</Step>
<Step title="Register your webhook endpoint">

In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy the
signing secret. Your code verifies the signature and handles `checkout.completed`. See
[Webhooks](/concepts/webhooks).

</Step>
<Step title="Test it, then go live">

Point your calls at the keyless sandbox, `https://ringup.dev/api/tools` (or keep the prod URLs with
your `rk_test_` key). Call your agent and order: the test caller (`+14155550142`) is recognized and
pays on the saved Visa ending 5858; a first-time caller is texted a link and enters
`4111 1111 1111 1111`. Expect the confirmation read back and a test-mode `checkout.completed`. Swap
to the live URLs to go live.

</Step>
</Steps>
