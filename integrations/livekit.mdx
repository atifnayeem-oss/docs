Add card-on-file payments to a LiveKit agent two ways. **Hosted** transfers the SIP participant to a
Ringup payment line at payment time, so Ringup runs the whole payment. **In your agent** keeps the
caller in your agent and calls Ringup's tools from your own code (recognize the caller, charge the
saved card), with no transfer at all. A recognized caller never reads a card aloud, so no card data
touches your agent either way.

## Prerequisites

- A running LiveKit agent with SIP inbound configured (the caller arrives as a SIP participant).
- Your LiveKit server credentials, and a Ringup [API key](/authentication).

## Set up

<Tabs>
<Tab title="Using Dashboard">

Your agent creates a Checkout Session and transfers the SIP participant to it, like redirecting to a
hosted web checkout. Ringup takes payment on its own line and returns the caller if you ask it to.

<Note>
The transfer is code in your agent, so there is no dashboard variant of the wiring itself. Your
Ringup account side (API keys, the webhook endpoint, go-live) lives in the dashboard.
</Note>

<Steps>
<Step title="Recognize the caller at the start">

Call `identify` from your agent code at the greeting, not at payment (a plain HTTP call, like the
checkout call below). The caller's number is available the instant the call connects; for a
returning caller `identify` returns their name, so your agent greets them by name. The cards it
can pay with come later, from the Checkout Session.
See [Recognize at the start of the call](/concepts/identity).

</Step>
<Step title="Create a Checkout Session when the order is confirmed">

`POST` to Ringup's `create_checkout` endpoint with your Ringup API key. If `payment_required` is
`none`, do not transfer: finish the call normally. Otherwise transfer to the returned `transfer_to`.

```bash
POST https://api.ringup.dev/v1/create_checkout
Authorization: Bearer <YOUR_RINGUP_KEY>
Content-Type: application/json

{
  "merchant_id": "tonys-pizza",
  "caller_phone": "+14155550142",
  "order_id": "ord_7f21",                            // the merchant's order; the total is read from it
  "return_to": "sip:your-agent@your-sip-host",          // optional: transfer the caller back after payment
  "platform_call_id": "call_abc123",                 // optional: ties the payment to your call logs
  "success_message": "You are all set, your order will be ready shortly."  // optional (if no return_to)
}
```

```jsonc
{
  "checkout_session_id": "cs_x9f2",                  // the correlation token
  "payment_required": "required",                    // "required" | "optional" | "none"
  "amount_cents": 1900,                              // resolved from the order
  "currency": "USD",
  "amount_source": "order",                          // "order" | "request"
  "transfer_to": "sip:cs_x9f2@transfer.ringup.dev",  // null unless payment is required
  "expires_at": "2026-07-21T18:35:12Z",
  "cards": [],                                       // saved cards chargeable at this merchant
  "default_card_id": null,
  "metadata": null
}
```

</Step>
<Step title="Transfer the SIP participant">

Speak one bridge line, then transfer the caller's SIP participant to `transfer_to`, attaching the
`checkout_session_id` as a REFER header as a backstop:

```python
# after saying: "One moment, connecting you to secure payment."
await livekit_api.sip.transfer_sip_participant(
    room_name=room,
    participant_identity=caller_identity,
    transfer_to=checkout["transfer_to"],                 # sip:cs_x9f2@transfer.ringup.dev
    headers={"X-Session-Id": checkout["checkout_session_id"]},  # cs_x9f2
    play_dialtone=False,                                 # the bridge line covers the gap
)
```

</Step>
<Step title="Register your webhook endpoint">

In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy the
signing secret. Your code verifies the signature and handles `checkout.completed`. See
[Webhooks](/concepts/webhooks).

</Step>
<Step title="Test it, then go live">

Point your `POST` at the keyless sandbox, `https://ringup.dev/api/tools/create_checkout` (or keep
the prod URL with your `rk_test_` key), call your agent, and order: the test caller
(`+14155550142`) pays on the saved Visa ending 5858; a first-time caller is texted a link and
enters `4111 1111 1111 1111`. Expect the payment line to read a confirmation code and a test-mode
`checkout.completed` at your endpoint. Swap the URL back to the live endpoint to go live.

</Step>
</Steps>

</Tab>
<Tab title="Using Tools">

Keep the caller in your agent and call Ringup's tools straight from your agent code with plain HTTP.
Recognize the caller at the greeting, then charge their saved card when the order is confirmed. No
transfer, no payment line.

<Steps>
<Step title="Recognize the caller at the greeting">

Call `identify` at the greeting, not at payment. Pass `caller_phone`, which LiveKit gives you the
instant the SIP participant joins. For a returning caller it returns their name, so your agent
greets them by name and skips questions it already has answers to. An unknown caller comes back
`known: false`, which is an ordinary first call. See [Recognize at the start of the
call](/concepts/identity).

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/identify \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "caller_phone": "+14155550142" }'
```

```javascript Node
const caller = await fetch("https://api.ringup.dev/v1/identify", {
  method: "POST",
  headers: {
    "Authorization": "Bearer <YOUR_RINGUP_KEY>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ caller_phone: callerNumber }),
}).then((r) => r.json());
// caller.known tells you whether to greet by caller.consumer.first_name
```
</CodeGroup>

</Step>
<Step title="Open a Checkout Session when the order is confirmed">

`POST` to `create_checkout` with the merchant and the caller. Ringup resolves the total (from the
merchant's order when you pass `order_id`), applies the merchant's
[payment policy](/concepts/payment-policy), and returns `cards`: the caller's cards chargeable at
this merchant. On `payment_required: "none"` say nothing about payment and finish the call.

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/create_checkout \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
    "merchant_id": "tonys-pizza",
    "caller_phone": "+14155550142",
    "order_id": "ord_7f21"
  }'
```

```javascript Node
const checkout = await fetch("https://api.ringup.dev/v1/create_checkout", {
  method: "POST",
  headers: {
    "Authorization": "Bearer <YOUR_RINGUP_KEY>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    merchant_id: merchantId,
    caller_phone: callerNumber,
    order_id: orderId,           // or amount_cents, when there is no order
  }),
}).then((r) => r.json());
// checkout.payment_required === "none" means say nothing about payment
```
</CodeGroup>

</Step>
<Step title="Charge the card the caller picked">

Read one card from `cards` back by brand and last four with the amount. After the caller agrees,
`POST` to `charge` with the session and that card's `instrument_id`. Ringup charges it off-session
on the merchant's own processor. A decline comes back as a `402` in the standard
[error envelope](/api-reference/errors), carrying the sentence your agent can speak.

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/charge \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "checkout_session_id": "cs_x9f2", "instrument_id": "ins_4b7e" }'
```

```javascript Node
const res = await fetch("https://api.ringup.dev/v1/charge", {
  method: "POST",
  headers: {
    "Authorization": "Bearer <YOUR_RINGUP_KEY>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    checkout_session_id: checkout.checkout_session_id,
    instrument_id: card.instrument_id,   // the card the caller agreed to
  }),
});
if (!res.ok) { /* a decline is a 402: speak error.agent_message */ }
const charge = await res.json();
// read charge.order_id back only when charge.status === "completed"
```
</CodeGroup>

</Step>
<Step title="Text a link when there is no card to offer">

When `cards` is empty (or the charge declined), `POST` to `send_link`. The session is the only
input, and Ringup texts a secure pay link. Check `sent`: `false` means a duplicate send returned the
same link without texting again. See [Send Link](/concepts/send-link).

<CodeGroup>
```bash curl
curl -X POST https://api.ringup.dev/v1/send_link \
  -H "Authorization: Bearer <YOUR_RINGUP_KEY>" \
  -H "Content-Type: application/json" \
  -d '{ "checkout_session_id": "cs_x9f2" }'
```

```javascript Node
const link = await fetch("https://api.ringup.dev/v1/send_link", {
  method: "POST",
  headers: {
    "Authorization": "Bearer <YOUR_RINGUP_KEY>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ checkout_session_id: checkout.checkout_session_id }),
}).then((r) => r.json());
// link.sent === false means a duplicate send: no new text went out, so do not promise one
```
</CodeGroup>

</Step>
<Step title="Register your webhook endpoint">

In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy the
signing secret. Your code verifies the signature and handles `checkout.completed`. See
[Webhooks](/concepts/webhooks).

</Step>
<Step title="Test it, then go live">

Point your calls at the keyless sandbox, `https://ringup.dev/api/tools` (or keep the prod URLs with
your `rk_test_` key). Call your agent and order: the test caller (`+14155550142`) is recognized and
pays on the saved Visa ending 5858; a first-time caller is texted a link and enters
`4111 1111 1111 1111`. Expect the confirmation read back and a test-mode `checkout.completed`. Swap
to the live URLs to go live.

</Step>
</Steps>
