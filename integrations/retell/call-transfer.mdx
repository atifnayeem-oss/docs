This page is **call transfer** on Retell: at payment time the live call moves to a Ringup
payment line, like a hosted web checkout over the phone (see
[Checkout Session](/concepts/checkout-session)). Reach for it when you are standardizing on a
single integration across several platforms.

On Retell the whole integration is **two config objects on your agent**: a custom function
(`create_checkout`) pointed at Ringup, and a `transfer_call` tool whose destination is the
address the function returns. Retell fills the destination itself through a dynamic variable,
so the model never types, sees, or can garble the transfer address. You never assemble a SIP
URI or correlate a call.

This page is end to end: setup, the call, the transfer, and what happens after payment.

## How the pieces map to Retell

| Ringup piece | On Retell |
|---|---|
| `identify` | A custom function your agent calls at the greeting |
| Checkout Session | The `create_checkout` custom function pointed at Ringup |
| The transfer itself | Retell's `transfer_call` tool, destination filled from the function's response |
| The result | A `checkout.completed` webhook to your server |

## Prerequisites

- A Retell agent you already run, with a phone number that can transfer (see the warning
  below).
- Your Retell API key.
- A Ringup account and [API key](/authentication). Test mode works out of the box on
  a shared sandbox, so there is nothing to connect. See [Testing](/concepts/testing).

## Set up

This is the one-time setup. The fastest path is the Ringup dashboard, which registers
everything for you. If you prefer to own every object in your Retell account, create them by
hand; both end in the same place.

<Tabs>
<Tab title="Using the dashboard">

<Steps>
  <Step title="Connect your Retell agent">
    In the Ringup dashboard, open your merchant, then **Add payment**. Paste your Retell API
    key. Ringup uses the key once, to list your agents and register the objects below, and
    never stores it.
  </Step>
  <Step title="Pick the agent">
    Choose the agent that answers this merchant's number from the dropdown. Ringup registers
    the `identify` function (called at the greeting, so a returning caller is recognized and
    greeted by name), the `create_checkout` function, and the `transfer_to_payment` tool on
    it, wired exactly as the API tab shows, and gives you the one prompt line to add.
  </Step>
  <Step title="Rehearse before you flip live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, place an
    order, and pay with the test card. The whole flow runs in test mode against the sandbox;
    no real card, no real charge. See [Testing](/concepts/testing).
  </Step>
  <Step title="Go live">
    When the rehearsal call reads a confirmation code back, click **Go live**. The same
    objects now run on your production agent with your live key.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

First, recognize the caller at the start: register `identify` as a custom function and call
it at the greeting, not at payment. It takes no inputs; the caller's number rides a header
Retell fills server-side. For a returning caller it returns their name and saved card, so
your agent greets them by name and skips re-asking for anything recognition already provides.
See [Recognize at the start of the call](/concepts/identity).

Then add two entries to your agent's tools (Retell dashboard: Agent, then Functions; or
`general_tools` via the API). First, the custom function that creates the Checkout Session:

```jsonc
{
  "type": "custom",
  "name": "create_checkout",
  "description": "Create the checkout session for the confirmed order. Returns the payment transfer destination.",
  "url": "https://api.ringup.dev/v1/checkouts",
  "method": "POST",
  "args_at_root": true,
  "headers": {
    "Authorization": "Bearer <YOUR_RINGUP_KEY>",
    "X-Caller-Number": "{{user_number}}"        // filled by Retell, never by the model
  },
  "parameters": {
    "type": "object",
    "properties": {
      "amount_cents":    { "type": "number", "description": "Order total in cents" },
      "success_message": { "type": "string", "description": "One short warm closing line for this order, written from the conversation. Example: 'Your large pepperoni will be ready around 7:45, see you soon, Alex!'" },
      "order_id":        { "type": "string", "description": "Your order reference, if you have one" }
    },
    "required": ["amount_cents"]
  },
  "response_variables": { "ringup_transfer_uri": "transfer_to" },
  "speak_during_execution": true,
  "execution_message_type": "static_text",
  "execution_message_description": "One moment, putting your payment through now.",
  "speak_after_execution": true,
  "timeout_ms": 10000
}
```

Second, the transfer tool. Its destination is the dynamic variable the function just filled:

```jsonc
{
  "type": "transfer_call",
  "name": "transfer_to_payment",
  "description": "Transfer the caller to the secure payment line. Call this immediately after create_checkout returns.",
  "transfer_destination": { "type": "predefined", "number": "{{ringup_transfer_uri}}" },
  "transfer_option": {
    "type": "cold_transfer",
    "cold_transfer_mode": "sip_refer",
    "show_transferee_as_caller": true
  }
}
```

Then add one line to your agent's prompt:

> When the caller has confirmed their order, call create_checkout with the order total in
> cents, then immediately call transfer_to_payment and say nothing more.

That is the whole setup. The `execution_message_description` line is the bridge the caller
hears while the session is created and the call moves, so there is never dead air; keep it to
one short sentence.

<Note>
In test mode, point the function's `url` at the shared sandbox instead:
`https://ringup.dev/api/tools/create_checkout` (keyless, no `Authorization` header), and add
one more parameter the model always passes: `"transfer": { "type": "boolean", "description":
"Always pass true" }`. The tools work as written on your first call. See
[Testing](/concepts/testing) for the demo caller and test card.
</Note>

</Tab>
</Tabs>

<Warning>
Transfer needs a number Retell can move a call from. Numbers purchased through Retell and
numbers imported over an elastic SIP trunk both work; for an imported number, enable call
transfer (SIP REFER) on the trunk at your telephony provider, or the transfer is refused. A
call that reaches the agent by **dial-to-SIP-URI** (a registered call) cannot use Retell's
transfer at all; put agents that take payments on a real number.
</Warning>

## What happens when the tools fire

When your agent calls `create_checkout` and then `transfer_to_payment`, the handoff happens in
one motion:

<Steps>
  <Step title="Ringup creates the Checkout Session">
    The amount comes from the function arguments; the caller comes from the
    `X-Caller-Number` header Retell interpolates from the call, and the merchant from your
    key, never from model-typed values. The response carries `transfer_to`, the payment
    line's address with the session sealed inside it.
  </Step>
  <Step title="Retell moves the call">
    `response_variables` has already stored `transfer_to` in the destination variable, so the
    transfer tool dials it directly. The cold transfer hands the call off through your
    number's telephony provider, and your agent's part of the call is over; the caller hears
    only the bridge line.
  </Step>
  <Step title="The payment line takes it from there">
    It recognizes the caller, confirms the total, charges the saved card on the merchant's
    own payment processor, reads the confirmation code, and ends the call. A first-time
    caller with no saved card is texted a secure pay link instead.
  </Step>
</Steps>

<Accordion title="Why the session rides in the SIP URI (and what this protects you from)">
The Checkout Session travels inside the transfer address itself, as a signed token in the SIP
URI. On Retell's cold-transfer leg the new call is originated by your telephony provider, so
custom SIP headers set on the transfer tool do not arrive at the destination; the URI is the
one carrier that survives. Because the token also holds the caller's number captured at
session creation, recognition on the payment line never depends on caller ID surviving the
transfer either. You never see or handle any of this; it is why the integration stays two
copy-paste objects.
</Accordion>

## End the call in the merchant's words

The payment line always reads the confirmation code itself, so the code can never be garbled.
Everything after the code is yours, three ways, in order of precedence:

1. **Per call, zero code**: your agent fills `success_message` on the `create_checkout` call
   it already makes, from the conversation it is already having ("Your large pepperoni will be
   ready in about 20 minutes, see you soon, Alex!"). Nothing to build; it is one optional
   argument.
2. **Once, statically**: set `success_message` and `failure_message` as `query_params` on the
   function. Templates may use `{confirmation}`, `{order_id}`, `{amount}`, and `{name}`,
   filled at charge time from the session and the charge result, never by a model.
3. **Neither**: Ringup reads a short warm default.

`failure_message` is spoken if the charge fails, before the payment line offers a texted pay
link as the backup.

## After payment

Ringup charges, texts the caller a receipt, and ends the call with the confirmation code read
aloud. Your server reconciles from a `checkout.completed` webhook, which carries the Payment
(`payment_id`, your `order_id`, the confirmation). Match on the `order_id` you passed, store
the `payment_id`, and mark your order paid. See [Webhooks](/concepts/webhooks) for the event
catalog, signature verification, and idempotency.

## Validation status

<Check>
This recipe is validated end to end on live calls: order confirmed, Checkout Session created
mid-call, call cold-transferred over SIP REFER from an imported trunk number, the full session
token arrived intact in the transfer address, the caller was recognized, the saved test card
(VISA ending 5858) was charged in the sandbox, and the confirmation code was read back. The
recognition, charge, receipt, and webhook behavior is the same payment path Ringup runs
everywhere.
</Check>

Three Retell specifics the validation surfaced, handled for you: custom SIP headers do not
survive the transfer leg (which is why the session rides the URI), the transfer destination
accepts a dynamic variable so the address is filled server-side rather than model-typed, and a
dial-to-SIP-URI (registered) call cannot transfer at all, which is why payments belong on a
real number. The closing-line options above follow the standard `create_checkout`
contract; the live validation used the default closing.
