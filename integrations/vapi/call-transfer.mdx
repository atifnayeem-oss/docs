**Call transfer is the recommended way to add Ringup to a Vapi agent.** At payment time your agent
moves the live call to a Ringup payment line, like a hosted checkout over the phone, and Ringup runs
the whole payment. The whole integration is one tool: your agent calls `transfer_to_payment` with
the order total, and Ringup does the rest. You write no payment logic, and every improvement Ringup
ships to the payment conversation reaches your callers with no change on your side.

Two alternatives exist. [Native handoff](/integrations/vapi/native) keeps the caller in your agent's
own voice with no transfer; it is in **limited preview** while we harden its reliability, so start
here unless you have a specific reason. To keep the caller in your own agent and add payment as
plain tools instead (recognize the caller, charge the saved card, no transfer at all), use the
**In your agent** tab of [native handoff](/integrations/vapi/native).

## Prerequisites

- A Vapi agent with a phone number that can transfer to SIP (see the warning below).
- Your Vapi API key, and a Ringup [API key](/authentication).

## Set up

<Tabs>
<Tab title="Using the dashboard (hosted)">

<Steps>
  <Step title="Connect Vapi, once for your account">
    In the Ringup dashboard, open **Integrations**, then **Connect Vapi**, and paste your Vapi
    API key. Ringup uses it to list your agents and provision the tools, and never stores it.
    One connection covers every merchant you run.
  </Step>
  <Step title="Pick the agents that take payments">
    Select the agents that answer merchant calls. Ringup registers `identify` and
    `transfer_to_payment` on each and shows one prompt line to paste. No per-merchant step:
    the merchant resolves from the dialed number, and activates when it connects its processor.
  </Step>
  <Step title="Add your webhook endpoint">
    Under **Settings, then Webhooks**, add your HTTPS endpoint and copy the signing secret.
    Ringup posts a signed `checkout.completed` there for every payment. See
    [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Test on the rehearsal number, then go live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, order, and
    pay with the test card (Visa ending 5858): no real charge. When it reads a confirmation
    code back, click **Go live**.
  </Step>
</Steps>

</Tab>
<Tab title="Wiring it by hand (hosted)">

<Steps>
  <Step title="Recognize the caller at the greeting">
    Register `identify` as an apiRequest tool and call it at the greeting. It takes no inputs;
    for a returning caller it returns their name and saved card, so your agent greets them by
    name. For an unknown caller it returns `known: false` and your agent continues with no
    name: never invent, guess, or reuse one, and never claim to recognize a caller `identify`
    did not return. See [Recognize at the start of the call](/concepts/identity).
  </Step>
  <Step title="Add the transfer tool">
    Create one async function tool pointed at Ringup, and add one prompt line: *when the
    caller has confirmed their order, use `transfer_to_payment` with the order total in cents,
    then say nothing more.*

    ```jsonc
    {
      "type": "function",
      "async": true,
      "server": {
        "url": "https://api.ringup.dev/v1/integrations/vapi/transfer",
        "secret": "<YOUR_RINGUP_KEY>"          // sent to Ringup as x-vapi-secret
      },
      "function": {
        "name": "transfer_to_payment",
        "description": "Transfer the caller to the secure payment line to charge their saved card. Pass the order total in cents.",
        "parameters": {
          "type": "object",
          "properties": {
            "amount_cents":    { "type": "number", "description": "Order total in cents" },
            "success_message": { "type": "string", "description": "Optional warm closing line spoken after payment, e.g. 'Your large pepperoni will be ready around 7:45, see you soon!' Never include the caller's name." },
            "order_id":        { "type": "string", "description": "Your order reference, if you have one" }
          },
          "required": ["amount_cents"]
        }
      },
      "messages": [
        { "type": "request-start", "content": "One moment, putting your payment through now.", "blocking": true }
      ]
    }
    ```
  </Step>
  <Step title="Add your webhook endpoint">
    In the Ringup dashboard under **Settings, then Webhooks**, add your HTTPS endpoint and copy
    the signing secret. Your server verifies the signature and handles `checkout.completed`
    (`payment_id`, your `order_id`, the confirmation). See [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Test it, then go live">
    Point the tool's `server.url` at the keyless sandbox,
    `https://ringup.dev/api/transfer-destination`. Call your agent and order: the test caller
    (`+14155550142`) pays on the saved Visa ending 5858; a first-time caller is texted a link
    and enters `4111 1111 1111 1111`. Expect a confirmation code read back and a test-mode
    `checkout.completed`. Swap the URL back to the live endpoint to go live.
  </Step>
</Steps>

</Tab>
</Tabs>

<Warning>
Transfer needs a number that can hand a call to SIP: Vapi-native and SIP-trunk (BYOC) numbers
both can. A number **imported from a telephony provider** cannot, so put those agents on
[native handoff](/integrations/vapi/native) or move the number onto a SIP trunk. The dashboard
flags which path fits each agent when you connect.
</Warning>

## What Ringup guarantees during the transfer

<Check>
**The hand-off is never silent.** Before the leg switches, Ringup speaks a heads-up in your
agent's turn ("Just a sec, connecting you to our secure payment line now") and holds the
transfer until the sentence finishes.
</Check>

<Check>
**A dropped session reference cannot strand a payment.** The Checkout Session rides the
transfer two ways at once: in the transfer itself and in a short-lived server-side record
keyed to the caller's number. If either carrier fails, the other completes the payment.
</Check>

<Check>
**The confirmation is pushed, not polled.** The moment a payment settles, whether by saved
card on the call or a texted link paid mid-call, the payment line announces it instantly.
</Check>
