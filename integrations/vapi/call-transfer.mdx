Vapi supports two ways to add Ringup payments. This page is **call transfer**: at payment time
the live call moves to a Ringup payment line, like a hosted web checkout over the phone (see
[Checkout Session](/concepts/checkout-session)). Reach for it when you are standardizing on a
single integration across several platforms. The other way, [native handoff](/integrations/vapi/native),
keeps the caller in your agent's own voice and is the more seamless option on Vapi.

On Vapi the whole integration is **one tool**. Your agent registers `transfer_to_payment`,
points it at Ringup, and calls it with the order total. Ringup creates the Checkout Session and
moves the call itself, through Vapi's live call control. You never assemble a destination,
touch SIP, or handle the transfer.

This page is end to end: setup, the call, the transfer, and what happens after payment.

## How the pieces map to Vapi

| Ringup piece | On Vapi |
|---|---|
| `identify` | An apiRequest tool your agent calls at the greeting |
| Checkout Session + transfer | One async `transfer_to_payment` function tool pointed at Ringup |
| The transfer itself | Ringup executes it through the call's live control channel |
| The result | A `checkout.completed` webhook to your server |

## Prerequisites

- A Vapi agent you already run, with a phone number.
- Your Vapi API key.
- A Ringup account and [API key](/authentication). Test mode works out of the box on
  a shared sandbox, so there is nothing to connect. See [Testing](/concepts/testing).

## Set up

This is the one-time setup. The fastest path is the Ringup dashboard, which registers both
tools for you. If you prefer to own every object in your Vapi account, create them by hand;
both end in the same place.

<Tabs>
<Tab title="Using the dashboard">

<Steps>
  <Step title="Connect your Vapi agent">
    In the Ringup dashboard, open your merchant, then **Add payment**. Paste your Vapi API
    key. Ringup uses the key once, to list your agents and register the tools, and never
    stores it.
  </Step>
  <Step title="Pick the agent">
    Choose the agent that answers this merchant's number from the dropdown. Ringup registers
    the `identify` tool (called at the greeting, so a returning caller is recognized and
    greeted by name) and the `transfer_to_payment` tool on it, wired exactly as the API tab
    shows, and gives you the one prompt line to add.
  </Step>
  <Step title="Rehearse before you flip live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, place an
    order, and pay with the test card. The whole flow runs in test mode against the sandbox;
    no real card, no real charge. See [Testing](/concepts/testing).
  </Step>
  <Step title="Go live">
    When the rehearsal call reads a confirmation code back, click **Go live**. The same
    tools now run on your production agent with your live key.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

First, recognize the caller at the start: register `identify` as an apiRequest tool and call
it at the greeting, not at payment. It takes no inputs; the caller's number comes from the
call. For a returning caller it returns their name and saved card, so your agent greets them
by name and skips re-asking for anything recognition already provides. See
[Recognize at the start of the call](/concepts/identity).

Then add the transfer tool: create one **async function tool** whose server URL points at
Ringup, and attach it to your agent (Vapi dashboard: Tools, then Create Tool, then Function;
or via the API):

```jsonc
{
  "type": "function",
  "async": true,
  "server": {
    "url": "https://api.ringup.dev/v1/integrations/vapi/transfer",
    "secret": "<YOUR_RINGUP_KEY>"          // sent to Ringup as x-vapi-secret
  },
  "function": {
    "name": "transfer_to_payment",
    "description": "Transfer the caller to the secure payment line to charge their saved card for the confirmed order. Pass the order total in cents.",
    "parameters": {
      "type": "object",
      "properties": {
        "amount_cents":    { "type": "number", "description": "Order total in cents" },
        "success_message": { "type": "string", "description": "One short warm closing line for this order, written from the conversation. Example: 'Your large pepperoni will be ready around 7:45, see you soon, Alex!'" },
        "order_id":        { "type": "string", "description": "Your order reference, if you have one" }
      },
      "required": ["amount_cents"]
    }
  },
  "messages": [
    { "type": "request-start", "content": "One moment, putting your payment through now.", "blocking": true }
  ]
}
```

Then add one line to your agent's prompt:

> When the caller has confirmed their order, use transfer_to_payment with the order total in
> cents, then say nothing more.

That is the whole setup. The `request-start` line is the bridge the caller hears while the call
moves, so there is never dead air; keep it to one short sentence.

<Note>
In test mode, point `server.url` at the shared sandbox instead:
`https://ringup.dev/api/transfer-destination` (keyless). The tool works as written on your
first call. See [Testing](/concepts/testing) for the demo caller and test card.
</Note>

</Tab>
</Tabs>

<Warning>
Transfer needs a number that can hand a call to SIP. Vapi-native numbers and SIP-trunk
(BYOC) numbers both can. A number **imported from a telephony provider** cannot move its
caller to SIP mid-call, so agents on imported numbers should use
[native handoff](/integrations/vapi/native) instead, or move the number onto a SIP trunk.
The dashboard checks this for you when you connect and tells you which path fits each agent.
</Warning>

## What happens when the tool fires

When your agent calls `transfer_to_payment`, Vapi sends the tool call to Ringup, and Ringup does
the rest in one motion:

<Steps>
  <Step title="Ringup creates the Checkout Session">
    The amount comes from the tool arguments; the caller and the merchant come from the call
    itself (the caller's number and your agent's dialed number), never from model-typed values.
  </Step>
  <Step title="Ringup moves the call">
    Ringup posts a transfer to the call's live control channel, and Vapi transfers the caller
    to the Ringup payment line. Your agent's part of the call is over; the caller hears only
    the bridge line.
  </Step>
  <Step title="The payment line takes it from there">
    It recognizes the caller, confirms the total, charges the saved card on the merchant's own
    payment processor, reads the confirmation code, and ends the call. A first-time caller with
    no saved card is texted a secure pay link instead.
  </Step>
</Steps>

<Accordion title="Why the session rides in the SIP URI (and what this protects you from)">
The Checkout Session travels inside the transfer address itself, as a signed token in the SIP
URI. On Vapi's transfer leg, custom SIP headers do not arrive at the destination and the
caller ID is not usable for identification, so the URI is the one carrier that survives.
Because the token also holds the caller's number captured at session creation, recognition on
the payment line never depends on caller ID surviving the transfer. You never see or handle
any of this; it is why the integration stays one tool.
</Accordion>

## End the call in the merchant's words

The payment line always reads the confirmation code itself, so the code can never be garbled.
Everything after the code is yours, three ways, in order of precedence:

1. **Per call, zero code**: your agent fills `success_message` on the tool call it already
   makes, from the conversation it is already having ("Your large pepperoni will be ready in
   about 20 minutes, see you soon, Alex!"). Nothing to build; it is one optional argument.
2. **Once, statically**: set `success_message` and `failure_message` as query parameters on the
   tool's server URL. Templates may use `{confirmation}`, `{order_id}`, `{amount}`, and
   `{name}`, filled at charge time from the session and the charge result, never by a model.
3. **Neither**: Ringup reads a short warm default.

`failure_message` (URL parameter) is spoken if the charge fails, before the payment line
offers a texted pay link as the backup.

## After payment

Ringup charges, texts the caller a receipt, and ends the call with the confirmation code read
aloud. Your server reconciles from a `checkout.completed` webhook, which carries the Payment
(`payment_id`, your `order_id`, the confirmation). Match on the `order_id` you passed, store
the `payment_id`, and mark your order paid. See [Webhooks](/concepts/webhooks) for the event
catalog, signature verification, and idempotency.

## Validation status

<Check>
This recipe is validated end to end on a live call: order taken, call transferred silently,
saved card charged in the sandbox, confirmation read back, webhook-ready result. The
recognition, charge, receipt, and webhook behavior is the same payment path Ringup runs
everywhere.
</Check>

Two Vapi specifics the validation surfaced, handled for you: a transfer destination cannot be
templated into a static transfer tool (Vapi rejects it), and custom SIP headers do not survive
the transfer leg. Both are why the integration is a server-driven tool rather than
configuration you assemble.
