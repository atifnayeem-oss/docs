Add card-on-file payments to a Vapi agent two ways. **Hosted** runs the payment step inside your
squad in your agent's own voice, so you write no payment prompt; set it up from the dashboard. **In
your agent** keeps the caller in your agent the whole call: you add three tools that recognize the
caller and charge their saved card, and nothing hands off. A recognized caller never reads a card
aloud, so no card data touches your agent either way. To standardize one integration across several
platforms instead, use [call transfer](/integrations/vapi/call-transfer).

## Prerequisites

- A Vapi agent (or squad) you already run.
- Your Vapi API key (private key), and a Ringup [API key](/authentication).

## Set up

<Tabs>
<Tab title="Using the dashboard (hosted)">

Ringup runs the payment step inside your squad, in your agent's own voice. Best when you want the
payment moment handled for you.

<Steps>
  <Step title="Connect Vapi, once for your account">
    In the Ringup dashboard, open **Integrations**, then **Connect Vapi**, and paste your Vapi
    API key (private key). Ringup uses it to list your agents and provision the payment objects,
    and never stores it. One connection covers every merchant you run.
  </Step>
  <Step title="Pick the agents that take payments">
    Select the agent that answers merchant calls (or the squad plus the member that confirms the
    order). Ringup assembles the payment step, the wrap-up step, the `ringup_*` tools, and the
    handoff tool around it, then shows the one prompt line to paste. No per-merchant step: the
    merchant resolves from the dialed number, and activates when it connects its processor.
  </Step>
  <Step title="Add your webhook endpoint">
    Under **Settings, then Webhooks**, add your HTTPS endpoint and copy the signing secret.
    Ringup posts a signed `checkout.completed` there for every payment. See
    [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Rehearse on the rehearsal number, then go live">
    The dashboard binds a test twin of your agent, with the go-live changes baked in, to a
    rehearsal number. Call it, order, and pay with the test card (Visa ending 5858): no real
    charge. When it reads a confirmation code back, click **Go live**. Ringup points your number
    at the squad, then makes exactly two changes to your assistant (appends the handoff tool and
    the one prompt line). **Roll back** any time restores it byte-for-byte.
  </Step>
</Steps>

<Warning>
Vapi's dashboard does not render squad-scoped wiring or apiRequest tool names; the squad canvas
may show "No handoff tools configured" for working wiring. Read the squad and assistant back from
the Vapi API to verify: it is the source of truth. Also, member `assistantOverrides` cannot
deliver tools on voice calls (toolIds are ignored, inline tools reach chat only), so the saved
assistant is the only wiring that works on voice.
</Warning>

</Tab>
<Tab title="In your agent (API)">

Keep the caller in your agent and add three Vapi `apiRequest` tools pointed at Ringup, authed with
your Ringup key. On Vapi the caller's number resolves server-side from the tool call, so you pass no
identity header and the model never types identity.

<Steps>
  <Step title="Recognize the caller at the greeting">
    Register `identify` as an apiRequest tool and call it at the greeting, not at payment. It takes
    no inputs; for a returning caller it returns their name and saved card, so your agent greets them
    by name and knows a saved card is on file. See
    [Recognize at the start of the call](/concepts/identity).

    ```jsonc
    {
      "type": "apiRequest",
      "name": "identify",
      "function": {
        "name": "identify",
        "description": "Recognize the caller at the greeting. Returns their name and whether a saved card is on file."
      },
      "url": "https://api.ringup.dev/v1/identify",
      "method": "POST",
      "headers": { "type": "object", "properties": {
        "Authorization": { "type": "string", "value": "Bearer <YOUR_RINGUP_KEY>" }
      } }
    }
    ```
  </Step>
  <Step title="Charge the saved card at payment">
    Register `charge`. When the order is confirmed and `identify` returned a saved card, your agent
    calls it with the order total; Ringup charges the saved card off-session on the merchant's own
    processor and returns the confirmation to read back. It respects the merchant's payment policy
    server-side, so on an unconnected merchant it is a silent no-op.

    ```jsonc
    {
      "type": "apiRequest",
      "name": "charge",
      "function": {
        "name": "charge",
        "description": "Charge the recognized caller's saved card for the confirmed order. Returns the confirmation code to read back.",
        "parameters": {
          "type": "object",
          "properties": {
            "amount_cents": { "type": "number", "description": "Order total in cents" },
            "order_id":     { "type": "string", "description": "Your order id, if you have one" }
          },
          "required": ["amount_cents"]
        }
      },
      "url": "https://api.ringup.dev/v1/charge",
      "method": "POST",
      "headers": { "type": "object", "properties": {
        "Authorization": { "type": "string", "value": "Bearer <YOUR_RINGUP_KEY>" }
      } }
    }
    ```
  </Step>
  <Step title="Text a link for a first-time caller">
    Register `send_link` for a caller with no saved card (or a declined charge). Ringup texts a
    secure pay link and, when they pay, pushes the confirmation back into the live call (or sends it
    by SMS if the call has ended).

    ```jsonc
    {
      "type": "apiRequest",
      "name": "send_link",
      "function": {
        "name": "send_link",
        "description": "Text the caller a secure pay link for the order. Use when there is no saved card on file.",
        "parameters": {
          "type": "object",
          "properties": {
            "amount_cents": { "type": "number", "description": "Order total in cents" }
          },
          "required": ["amount_cents"]
        }
      },
      "url": "https://api.ringup.dev/v1/send_link",
      "method": "POST",
      "headers": { "type": "object", "properties": {
        "Authorization": { "type": "string", "value": "Bearer <YOUR_RINGUP_KEY>" }
      } }
    }
    ```

    Then add one line to your prompt: *when the caller has confirmed the order, if identify returned
    a saved card call `charge` with the order total and read back the confirmation code; otherwise
    call `send_link` and tell them to check their text to pay.*
  </Step>
  <Step title="Add your webhook endpoint">
    In the Ringup dashboard under **Settings, then Webhooks**, add your HTTPS endpoint and copy the
    signing secret. Your server verifies the signature and handles `checkout.completed`. See
    [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Test it, then go live">
    Point the tools' `url` at the keyless sandbox, `https://ringup.dev/api/tools` (or keep the prod
    URLs with your `rk_test_` key). Call your agent and order: the test caller (`+14155550142`) is
    recognized and pays on the saved Visa ending 5858; a first-time caller is texted a link and
    enters `4111 1111 1111 1111`. Expect the confirmation read back and a test-mode
    `checkout.completed`. Swap to the live URLs to go live.
  </Step>
</Steps>

</Tab>
