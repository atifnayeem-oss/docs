Add card-on-file payments to an ElevenLabs agent by call transfer: when the order is confirmed,
your agent registers a **Checkout Session** and transfers the live call to a standing Ringup
payment address, like redirecting to a hosted web checkout. ElevenLabs pins the transfer
destination in agent config at call start (it ignores model-typed destinations and mid-call
edits), so the address is set once at setup and `create_checkout` registers each order mid-call;
Ringup matches the transferred call to that order by the caller's number. If the model is new to
you, read [Call transfer](/concepts/checkout-session) first.

## Prerequisites

- An ElevenLabs agent on a **SIP-trunk number** (see the warning below).
- Your ElevenLabs API key, and a Ringup [API key](/authentication).

## Set up

<Warning>
**Preview.** The transfer transport is live: an agent on a SIP-trunk number can refer the call via
SIP REFER to the payment line. The piece that makes this work for real per-order amounts, matching
the transferred call to the order you registered by the caller's number, is still in build.
Confirm availability in your Ringup dashboard before relying on this in production.
</Warning>

<Tabs>
<Tab title="Using the dashboard">

<Steps>
  <Step title="Connect ElevenLabs, once for your account">
    In the Ringup dashboard, open **Integrations**, then **Connect ElevenLabs**, and paste your
    ElevenLabs API key. Ringup uses it to list your agents and provision the payment objects, and
    never stores it. One connection covers every merchant you run.
  </Step>
  <Step title="Pick the agents that take payments">
    Select the agents that answer merchant calls. Ringup registers the `identify` and
    `create_checkout` tools on each and pins your payment address in the agent's transfer rule;
    paste the one prompt line the dashboard shows. No per-merchant step: the merchant resolves from
    the dialed number, and activates when it connects its processor.
  </Step>
  <Step title="Add your webhook endpoint">
    Under **Settings, then Webhooks**, add your HTTPS endpoint and copy the signing secret. Ringup
    posts a signed `checkout.completed` there for every payment. See [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Rehearse on the rehearsal number, then go live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, order, and pay
    with the test card (Visa ending 5858): no real charge. When it reads a confirmation code back,
    click **Go live**.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

<Steps>
  <Step title="Recognize the caller at the greeting">
    Register `identify` as a webhook tool (it takes no arguments; the caller's number comes from
    the call) and call it at the greeting, not at payment. For a returning caller it returns their
    name and saved card, so your agent greets them by name. See
    [Recognize at the start of the call](/concepts/identity).
  </Step>
  <Step title="Add the Ringup checkout tool">
    On your agent, add a **webhook tool** that calls Ringup's `create_checkout` with your Ringup
    API key (Agent, then Tools, then Add tool, then Webhook), and add one prompt line: *when the
    caller has confirmed their order, call `create_checkout` with the order total, then transfer to
    payment unless it returns `payment_required: "none"`.*

    ```jsonc
    {
      "type": "webhook",
      "name": "create_checkout",
      "description": "Register the confirmed order for payment and learn whether payment is required.",
      "url": "https://api.ringup.dev/v1/checkouts",
      "method": "POST",
      "request_headers": { "Authorization": "Bearer <YOUR_RINGUP_KEY>" },
      "body_parameters": {
        "amount_cents": { "type": "number", "description": "Order total in cents" },
        "order_id":     { "type": "string", "description": "Your processor order id, if you have one" }
      }
    }
    ```
  </Step>
  <Step title="Pin the transfer rule">
    Add the `transfer_to_number` system tool with your standing Ringup payment address (from your
    dashboard) as a SIP-URI destination and `transfer_type: sip_refer`:

    ```jsonc
    // conversation_config.agent.prompt.built_in_tools.transfer_to_number
    {
      "type": "system",
      "name": "transfer_to_number",
      "description": "",
      "params": {
        "system_tool_type": "transfer_to_number",
        "transfers": [{
          "transfer_destination": { "type": "sip_uri", "sip_uri": "<YOUR_RINGUP_PAYMENT_ADDRESS>" },
          "transfer_type": "sip_refer",
          "condition": "When the caller has confirmed the order and create_checkout says payment is required or offered."
        }]
      }
    }
    ```

    The destination is resolved from this rule when the call starts, which is why it is standing
    config, not a per-order value.
  </Step>
  <Step title="Add your webhook endpoint">
    In the Ringup dashboard, under **Settings, then Webhooks**, add your HTTPS endpoint and copy
    the signing secret. Your server verifies the signature and handles `checkout.completed`. See
    [Webhooks](/concepts/webhooks).
  </Step>
  <Step title="Test it, then go live">
    Point the `create_checkout` tool's `url` at the keyless sandbox,
    `https://ringup.dev/api/tools/create_checkout` (or keep the prod URL with your `rk_test_` key).
    Call your agent and order: the test caller (`+14155550142`) pays on the saved Visa ending 5858;
    a first-time caller is texted a link and enters `4111 1111 1111 1111`. Expect a confirmation
    code read back and a test-mode `checkout.completed`. Swap the URL back to the live endpoint to
    go live.
  </Step>
</Steps>

</Tab>
</Tabs>

<Warning>
The agent's number must be attached over a **SIP trunk**. ElevenLabs allows a SIP-URI transfer
destination only with `sip_refer`, and rejects that combination on a native-Twilio number at
configuration time, so an agent on a native-Twilio number cannot reach the payment line by any
mode. Custom SIP headers also do not survive the transfer, which is why correlation runs on the
registered session and the caller's number, never on headers.
</Warning>
