Add card-on-file payments to an agent built on the OpenAI Realtime API's SIP connector. This
works like a hosted web checkout, over the phone: when the order is confirmed, your server
creates a **Checkout Session** and transfers the live call to a Ringup payment line with a single
REST call, the caller pays there, and the call ends with a confirmation read aloud. Ringup
recognizes the caller, charges their saved card, settles it against the order, and texts a
receipt. Your server reconciles from a webhook. If the model is new to you, read
[Call transfer](/concepts/checkout-session) first.

On OpenAI Realtime there is no hosted agent and no platform tool to register. You already run the
server that answers the call (the webhook handler that accepts `realtime.call.incoming` and opens
the session). The Ringup integration is two REST calls from that same server: create the Checkout
Session, then `refer` the call to the SIP URI it hands back. There is no function tool to install.

This page is end to end: setup, the call, the transfer, and what happens after payment.

## How the pieces map to OpenAI Realtime

| Ringup piece | On OpenAI Realtime |
|---|---|
| `identify` | An HTTP `POST` your server makes at the greeting, over the session's control WebSocket |
| `create_checkout` | An HTTP `POST` your server makes when the order is confirmed |
| The transfer target | The `transfer_to` SIP URI the Checkout Session returns |
| The transfer | `POST /v1/realtime/calls/{call_id}/refer` with `target_uri` set to `transfer_to` |
| The result | A `checkout.completed` webhook to your server |

## Prerequisites

- An OpenAI Realtime SIP agent already answering calls: a SIP trunk pointed at
  `sip:$PROJECT_ID@sip.api.openai.com`, and a project webhook configured to receive
  `realtime.call.incoming` so your server can accept the call and hold the `call_id`.
- Your OpenAI API key (the `call_id` and the `refer` call are authenticated with it).
- A Ringup account and [API key](/authentication). Test mode works out of the box on a shared
  sandbox. See [Testing](/concepts/testing).

## Set up

<Tabs>
<Tab title="Using the dashboard">

On OpenAI Realtime the wiring is code in your server, so the dashboard path covers the account
side: connecting the merchant, rehearsing in test mode, and going live. The calls themselves are
written in your server either way; they are the ones in the **Using the API** tab.

<Steps>
  <Step title="Add the payment">
    In the Ringup dashboard, open your merchant, then **Add payment**, and pick OpenAI Realtime.
  </Step>
  <Step title="Wire your server">
    Add the identify call, the Checkout Session, and the refer from the **Using the API** tab.
    The wiring lives in your server code, so it is identical on both paths.
  </Step>
  <Step title="Rehearse before you flip live">
    With the merchant in test mode, call your agent, place an order, and pay with the test card.
    The whole flow runs against the sandbox: no real card, no real charge. See
    [Testing](/concepts/testing).
  </Step>
  <Step title="Go live">
    When the rehearsal call reads a confirmation code back, click **Go live**.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

<Steps>
<Step title="Recognize the caller at the start">

Call `identify` from your server at the greeting, not just at payment. It needs only the caller's
phone number, which arrives in the `realtime.call.incoming` webhook as the SIP `From` header, so
it is available the instant the call connects. For a returning caller it returns their name and
saved card, so your agent greets them by name and skips re-asking for anything recognition already
provides (for example, never ask "what name for the order?" when `identify` returned it). This is
separate from and happens earlier than creating the Checkout Session and transferring for payment.
See [Recognize at the start of the call](/concepts/identity).

</Step>
<Step title="Create a Checkout Session on your server">

When the order is confirmed, `POST` to Ringup's `create_checkout` endpoint with your Ringup API
key. There is no platform tool to register; this is a plain HTTP call from the server that already
holds the `call_id`.

```bash
POST https://api.ringup.dev/v1/checkouts
Authorization: Bearer <YOUR_RINGUP_KEY>
Content-Type: application/json

{
  "amount_cents": 2500,
  "caller": "+14155551234",                          // the SIP From from the incoming webhook
  "line_items": [{ "name": "Large pepperoni", "quantity": 1, "amount_cents": 2500 }], // optional
  "order_id": "order_abc",                            // optional: settle against an existing order
  "success_message": "You are all set, your order will be ready shortly."  // optional
}
```

Response (the Checkout Session):

```jsonc
{
  "id": "cs_x9f2",                                    // Checkout Session id; the correlation token
  "payment_required": "required",                     // "required" | "optional" | "none"
  "transfer_to": "sip:<signed-token>@transfer.ringup.dev"  // where to REFER; the token embeds the id and caller
}
```

If `payment_required` is `none`, do not transfer: finish the call normally. Otherwise transfer.

<Note>
The Checkout Session carries the caller's number, captured here at creation, so recognition on the
payment line never depends on caller ID surviving the transfer. Pass `caller` from the incoming
webhook's SIP `From`; do not rely on the transferred leg to carry it.
</Note>

</Step>
<Step title="REFER the call to the payment line">

Speak one bridge line over the session, then `refer` the call to `transfer_to`. The target is the
full SIP URI the Checkout Session returned, unchanged:

```bash
POST https://api.openai.com/v1/realtime/calls/{call_id}/refer
Authorization: Bearer <YOUR_OPENAI_KEY>
Content-Type: application/json

{
  "target_uri": "sip:<signed-token>@transfer.ringup.dev"   // exactly the transfer_to from the session
}
```

OpenAI hands a SIP REFER to your trunk, which places the caller on the Ringup payment line. Your
agent's part of the call is over; the caller hears only the bridge line and then the payment line.

<Warning>
OpenAI's `refer` carries `target_uri` and nothing else: no custom SIP headers, no caller-ID
control. That is why the whole Checkout Session rides inside the SIP URI itself, as a signed token
in the user part. Send `transfer_to` verbatim; do not try to attach an `X-Session-Id` header,
because it would not survive. Correlation holds on the token alone.
</Warning>

A `200` from `refer` means the REFER was relayed to your trunk, not that the far end answered.
Because it is a blind transfer, OpenAI leaves the call once the REFER is relayed. Wrap the call in
a timeout and, if it fails, fall back to a texted pay link with [`send_link`](/concepts/send-link).

</Step>
</Steps>

## The result (the webhook)

Ringup answers the transferred call, recognizes the caller, charges the saved card, reads the
confirmation code aloud, and texts a receipt. A first-time caller with no saved card is texted a
secure pay link instead. Everything after the confirmation code is yours: pass `success_message`
(and `failure_message`) on `create_checkout`, or let Ringup read a short default. The receipt is
the proof either way.

The transfer is blind, so the caller is not returned to your OpenAI agent after payment. Reconcile
on your server from a webhook:

```jsonc
{
  "id": "evt_9f2",
  "type": "checkout.completed",         // or "checkout.failed"
  "data": {
    "checkout_id": "cs_x9f2",
    "payment_id": "pay_abc",
    "amount_cents": 2500,
    "order_id": "order_abc",            // echoed back
    "confirmation": "2PJJZY",
    "caller": "+14155551234",
    "card": { "brand": "VISA", "last_4": "5858" }
  }
}
```

Match on the `order_id` you passed, store the `payment_id`, and flip your order to paid. See
[Webhooks](/concepts/webhooks) for the full catalog, statuses, signature verification, and
idempotency.

## Validation status

<Warning>
This recipe is **documented, not yet validated end to end**. The transfer contract is confirmed
against OpenAI's current Realtime SIP reference (`POST /v1/realtime/calls/{call_id}/refer`, body
`target_uri`, `sip:` or `tel:` accepted), the Checkout Session mints a `transfer_to` SIP URI as
shown, and OpenAI's `refer` endpoint imposes no length or format limit on `target_uri` that would
reject the signed token. What is not yet confirmed on a live call is the final hop: the token URI
surviving the SIP REFER across the trunk and the caller landing on the Ringup payment line with
the session intact. Treat this page as the intended shape and confirm on a rehearsal call before
relying on it in production.
</Warning>

Two OpenAI specifics shape the integration, both handled above: `refer` is a blind transfer, so
there is no automatic return to your agent after payment, and it carries no custom SIP headers, so
the Checkout Session travels inside the SIP URI rather than a header. This is why the token is
self-describing and why recognition never depends on caller ID.
