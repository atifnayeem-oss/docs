This page is **call transfer** on ElevenLabs Agents: at payment time the live call moves to a
Ringup payment line, like a hosted web checkout over the phone (see
[Checkout Session](/concepts/checkout-session)). Reach for it when you are standardizing on a
single integration across several platforms.

On ElevenLabs the whole integration is **three config objects on your agent**: an `identify`
webhook tool called at the greeting, a `create_checkout` webhook tool pointed at Ringup, and a
`transfer_to_number` rule whose destination is your merchant's payment address. ElevenLabs pins
the transfer destination in agent config, so the model never chooses, types, or can garble the
transfer address. You never assemble a SIP URI or correlate a call.

This page is end to end: setup, the call, the transfer, and what happens after payment.

## How the pieces map to ElevenLabs

| Ringup piece | On ElevenLabs |
|---|---|
| `identify` | A webhook tool your agent calls at the greeting |
| Checkout Session | The `create_checkout` webhook tool pointed at Ringup |
| The transfer itself | The built-in `transfer_to_number` system tool, SIP REFER to your merchant's payment address |
| The result | A `checkout.completed` webhook to your server |

## Prerequisites

- An ElevenLabs agent you already run, on a **SIP trunk phone number** (see the warning below).
- Your ElevenLabs API key.
- A Ringup account and [API key](/authentication). Test mode works out of the box on
  a shared sandbox, so there is nothing to connect. See [Testing](/concepts/testing).

## Set up

This is the one-time setup. The fastest path is the Ringup dashboard, which registers
everything for you. If you prefer to own every object in your ElevenLabs workspace, create them
by hand; both end in the same place.

<Tabs>
<Tab title="Using the dashboard">

<Steps>
  <Step title="Connect your ElevenLabs agent">
    In the Ringup dashboard, open your merchant, then **Add payment**. Paste your ElevenLabs
    API key. Ringup uses the key once, to list your agents and register the objects below, and
    never stores it.
  </Step>
  <Step title="Pick the agent">
    Choose the agent that answers this merchant's number from the dropdown. Ringup registers
    the `identify` tool (called at the greeting, so a returning caller is recognized and
    greeted by name), the `create_checkout` tool, and the transfer rule on it, wired exactly
    as the API tab shows, and gives you the one prompt line to add.
  </Step>
  <Step title="Rehearse before you flip live">
    The dashboard binds a test twin of your agent to a rehearsal number. Call it, place an
    order, and pay with the test card. The whole flow runs in test mode against the sandbox;
    no real card, no real charge. See [Testing](/concepts/testing).
  </Step>
  <Step title="Go live">
    When the rehearsal call reads a confirmation code back, click **Go live**. The same
    objects now run on your production agent with your live key.
  </Step>
</Steps>

</Tab>
<Tab title="Using the API">

First, recognize the caller at the start: register `identify` as a webhook tool and have the
agent call it at the greeting, not at payment. The caller's number rides a field ElevenLabs
fills from the call itself (`system__caller_id`), never from model-typed text. For a returning
caller it returns their name and saved card, so your agent greets them by name and skips
re-asking for anything recognition already provides. See
[Recognize at the start of the call](/concepts/identity).

Then create the Checkout Session tool (`POST /v1/convai/tools` on the ElevenLabs API, or
Agent, then Tools in their dashboard):

```jsonc
{
  "tool_config": {
    "type": "webhook",
    "name": "create_checkout",
    "description": "Create the checkout session for the confirmed order. Call this with the order total in cents when the caller confirms they want to pay.",
    "response_timeout_secs": 15,
    "api_schema": {
      "url": "https://api.ringup.dev/v1/checkouts",
      "method": "POST",
      "request_headers": { "Authorization": "Bearer <YOUR_RINGUP_KEY>" },
      "request_body_schema": {
        "type": "object",
        "required": ["amount_cents"],
        "properties": {
          "amount_cents":    { "type": "integer", "description": "Order total in cents" },
          "phone":           { "type": "string", "dynamic_variable": "system__caller_id", "description": "" },
          "success_message": { "type": "string", "description": "One short warm closing line for this order, written from the conversation. Example: 'Your large pepperoni will be ready around 7:45, see you soon, Alex!'" },
          "order_id":        { "type": "string", "description": "Your order reference, if you have one" }
        }
      }
    }
  }
}
```

The `phone` field is filled by ElevenLabs from the live call (`dynamic_variable`), so the
model never sees or types it. Attach the returned tool id to your agent's
`conversation_config.agent.prompt.tool_ids` (keep `tool_ids`, not the deprecated inline
`tools` array).

Second, the transfer rule. It is ElevenLabs' built-in `transfer_to_number` system tool, and
its destination is your **merchant payment address** from the Ringup dashboard, set once
(`PATCH /v1/convai/agents/{agent_id}`, under
`conversation_config.agent.prompt.built_in_tools`):

```jsonc
{
  "transfer_to_number": {
    "type": "system",
    "name": "transfer_to_number",
    "description": "",
    "params": {
      "system_tool_type": "transfer_to_number",
      "transfers": [{
        "transfer_destination": { "type": "sip_uri", "sip_uri": "<YOUR_MERCHANT_PAYMENT_ADDRESS>" },
        "transfer_type": "sip_refer",
        "condition": "The caller has confirmed the order and create_checkout has returned."
      }]
    }
  }
}
```

Then add one line to your agent's prompt:

> When the caller has confirmed their order, call create_checkout with the order total in
> cents, then immediately use transfer_to_number and say nothing more.

That is the whole setup. The short line the agent speaks at the handoff is the bridge the
caller hears while the call moves, so there is never dead air; ElevenLabs prompts the model
for it automatically.

<Note>
In test mode, point the tool's `url` at the shared sandbox instead:
`https://ringup.dev/api/tools/create_checkout` (keyless, no `Authorization` header), and add
one more parameter the model always passes: `"transfer": { "type": "boolean", "description":
"Always pass true" }`. The sandbox response carries a ready `transfer_to` address for your
rehearsal rule. See [Testing](/concepts/testing) for the demo caller and test card.
</Note>

</Tab>
</Tabs>

<Warning>
Transfer to the payment line needs a **SIP trunk number** on ElevenLabs, and only
`transfer_type: "sip_refer"` can carry a SIP destination; ElevenLabs enforces this at
configuration time ("SIP URI is supported only for SIP REFER transfer type"). A number
imported through the native Twilio integration cannot move its caller to SIP by any mode, so
agents on Twilio-native numbers must move onto a SIP trunk first. Your trunk must also allow
transfer via SIP REFER; on a Twilio Elastic SIP trunk that is the Call Transfer setting
(`TransferMode=enable-all`), and keep `TransferCallerId=from-transferee` so the payment line
sees the real caller. The dashboard checks all of this for you when you connect.
</Warning>

## What happens when the tools fire

When your agent calls `create_checkout` and then `transfer_to_number`, the handoff happens in
one motion:

<Steps>
  <Step title="Ringup creates the Checkout Session">
    The amount comes from the tool arguments; the caller comes from the field ElevenLabs
    fills from the call, and the merchant from your key, never from model-typed values. The
    session is registered against your merchant's payment address before the call moves.
  </Step>
  <Step title="ElevenLabs moves the call">
    The agent invokes the transfer rule, ElevenLabs sends a SIP REFER to your trunk, and your
    trunk dials the payment address as a fresh leg. Your agent's part of the call is over;
    the caller hears only the bridge line.
  </Step>
  <Step title="The payment line takes it from there">
    It recognizes the caller, confirms the total, charges the saved card on the merchant's
    own payment processor, reads the confirmation code, and ends the call. A first-time
    caller with no saved card is texted a secure pay link instead.
  </Step>
</Steps>

<Accordion title="Why the session rides in the SIP URI (and what this protects you from)">
The session travels inside the transfer address itself, as a signed token in the SIP URI. On
the REFER leg the new call is originated by your telephony provider, and custom SIP headers
set on the transfer (including the ones ElevenLabs adds itself) do not arrive at the
destination; the URI is the one carrier that survives. The destination is also pinned in
agent config: ElevenLabs ignores any address the model types into the tool call, so a caller
cannot talk your agent into transferring the payment somewhere else. Recognition on the
payment line never depends on caller ID surviving the transfer either. You never see or
handle any of this; it is why the integration stays three copy-paste objects.
</Accordion>

## End the call in the merchant's words

The payment line always reads the confirmation code itself, so the code can never be garbled.
Everything after the code is yours, three ways, in order of precedence:

1. **Per call, zero code**: your agent fills `success_message` on the `create_checkout` call
   it already makes, from the conversation it is already having ("Your large pepperoni will be
   ready in about 20 minutes, see you soon, Alex!"). Nothing to build; it is one optional
   argument.
2. **Once, statically**: set `success_message` and `failure_message` as query parameters on
   the tool's `url`. Templates may use `{confirmation}`, `{order_id}`, `{amount}`, and
   `{name}`, filled at charge time from the session and the charge result, never by a model.
3. **Neither**: Ringup reads a short warm default.

`failure_message` is spoken if the charge fails, before the payment line offers a texted pay
link as the backup.

## After payment

Ringup charges, texts the caller a receipt, and ends the call with the confirmation code read
aloud. Your server reconciles from a `checkout.completed` webhook, which carries the Payment
(`payment_id`, your `order_id`, the confirmation). Match on the `order_id` you passed, store
the `payment_id`, and mark your order paid. See [Webhooks](/concepts/webhooks) for the event
catalog, signature verification, and idempotency.

## Validation status

<Check>
The transfer path is validated end to end on live calls: an ElevenLabs agent on a SIP trunk
number created the Checkout Session mid-call through a webhook tool, moved the live call over
SIP REFER, the full session token arrived intact in the transfer address (including tokens
close to 400 characters), the real caller's number survived the handoff, the caller was
recognized, the saved test card (VISA ending 5858) was charged in the sandbox, and the
confirmation code was read back. The recognition, charge, receipt, and webhook behavior is the
same payment path Ringup runs everywhere.
</Check>

Four ElevenLabs specifics the validation surfaced, handled for you: only SIP REFER can carry a
SIP destination and ElevenLabs rejects anything else at configuration time; the destination is
pinned in agent config and model-typed addresses are ignored (which is why the address is set
at setup rather than filled per call, and why the payment line matches your session
server-side); custom SIP headers do not survive the transfer leg, which is why the session
rides the URI; and Twilio-native numbers cannot transfer to SIP at all, which is why payments
belong on a SIP trunk number. The live validation ran the rehearsal shape with a
sandbox-issued transfer address; the dashboard flow above issues your merchant's address the
same way.
